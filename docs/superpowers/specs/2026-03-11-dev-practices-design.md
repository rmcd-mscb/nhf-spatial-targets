# Development Best Practices Infrastructure

**Date:** 2026-03-11
**Status:** Approved

## Goal

Embed software development best practices into the nhf-spatial-targets repository so that all contributors — whether using Claude or not — follow a consistent workflow with automated quality gates.

## Design

### 1. Pre-commit hooks (`.pre-commit-config.yaml`)

Uses the pre-commit framework. All hooks use `repo: local` since they invoke pixi tasks. Runs on every commit for all developers.

**New pixi task required:** Add `fmt-check` task (`ruff format --check src/ tests/`) to `pixi.toml`.

**Add `pre-commit` to `[feature.dev.dependencies]`** so it is managed by pixi rather than relying on a system install.

**Hook sequence (all hooks use `pixi run -e dev`):**

1. `pixi install -e dev` — ensure environment is current (fast no-op when already up to date)
2. `pixi run -e dev fmt-check` — verify formatting (fails if unformatted)
3. `pixi run -e dev lint` — ruff lint check
4. `pixi run -e dev test -- -m "not integration"` — run unit tests only (skip `pytest.mark.integration` tests which may need network/data)

Developers install once after cloning: `pixi run -e dev pre-commit install`.

### 2. GitHub Actions CI (`.github/workflows/ci.yml`)

Runs on every PR targeting `main`. Runner: `ubuntu-latest`.

**Steps:**

1. Checkout repository
2. Install pixi (via `prefix-dev/setup-pixi` action)
3. `pixi run fmt-check` — assert formatting (same gate as pre-commit)
4. `pixi run lint`
5. `pixi run test` (full suite including integration tests)

CI runs the full test suite since it has controlled environment and network access. Note: CI uses `fmt-check` (assertion) not `fmt` (mutation) — `fmt` is only used in Claude's PostToolUse hook to auto-fix files in the working tree.

### 3. Branch protection for `main`

Configured via GitHub API (`gh api`) once remote is pushed:

- Require PR to merge (no direct push to main)
- Require CI status checks to pass before merge
- No force pushes allowed

Squash merge enforced separately via repository settings (`gh api` to set `allow_squash_merge: true`, `allow_merge_commit: false`, `allow_rebase_merge: false`).

### 4. Git flow workflow

1. Create a GitHub issue describing the work
2. Branch from main using pattern: `<type>/<issue#>-short-description`
   - Valid types: `feature`, `fix`, `refactor`, `docs`, `test`, `chore`
   - Example: `feature/12-add-mwbm-fetch`, `fix/13-catalog-validation`
3. Develop on the branch, committing as needed
4. Open PR referencing the issue (e.g., "Closes #12")
5. CI must pass; squash merge after review

### 5. Claude hooks update (`.claude/settings.json`)

**Changes to existing config** (adds to current PostToolUse lint hook, adds new PreToolUse):

**PostToolUse (Edit|Write):**

- Run `pixi run fmt` (auto-format, mutates files) — added before existing lint hook. This is Claude-only; pre-commit and CI use `fmt-check` (assertion) instead.
- Run `pixi run lint` (lint check) — already exists, keep as-is

**PreToolUse (Edit|Write) — new:**

- Block direct file edits (Edit/Write tool) to `.credentials.yml`, `data/raw/*`, `pixi.lock`
- `pixi.lock` is auto-generated by `pixi install`; this block prevents Claude from directly editing the lock file but does not interfere with `pixi install` regenerating it via the Bash tool

**Permissions — add to existing allow list:**

- `Bash(git branch*)`, `Bash(ls *)`, `Bash(wc *)`

### 6. CLAUDE.md additions

- Document branch naming convention and PR workflow
- Pre-commit checklist: "Run `pixi run fmt && pixi run lint && pixi run test` before suggesting a commit"
- Test coverage rule: "Every new module in `fetch/`, `aggregate/`, `normalize/`, or `targets/` must have a corresponding `tests/test_<module>.py`" (sharpens existing guidance into an enforceable rule)
- Note that `pixi run` is required for all commands (not raw python/pytest/ruff)

### 7. CONTRIBUTING.md

Short contributing guide for colleagues without Claude:

- Prerequisites (pixi)
- Setup steps (clone, `pixi install -e dev`, `pixi run -e dev pre-commit install`)
- Branch naming convention
- PR expectations and merge strategy

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Pre-commit framework | pre-commit (not raw git hooks) | Standard for Python, shareable, managed via pixi |
| Hook type | `repo: local` | Hooks invoke pixi tasks, not standalone tools |
| CI runner | `ubuntu-latest` only | Primary platform; matrix can be added later |
| Merge strategy | Squash merge | One commit per PR on main, clean audit trail |
| Branch naming | `<type>/<issue#>-desc` | Descriptive, consistent with hydro-param convention |
| pixi install in hooks | Yes | Slower first time but ensures env is always current |
| Integration tests in pre-commit | Excluded (`-m "not integration"`) | Too slow/fragile for local commits; CI runs full suite |

## Appendix: Branch Protection Setup

Run these commands after the repository is pushed to GitHub. Replace `OWNER/REPO`
with the actual GitHub owner and repository name.

```bash
# Enforce squash merge only
gh api repos/OWNER/REPO \
  -X PATCH \
  -f allow_squash_merge=true \
  -f allow_merge_commit=false \
  -f allow_rebase_merge=false

# Enable branch protection on main
gh api repos/OWNER/REPO/branches/main/protection \
  -X PUT \
  -H "Accept: application/vnd.github+json" \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": true,
    "contexts": ["check"]
  },
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "allow_force_pushes": false,
  "allow_deletions": false
}
EOF
```
