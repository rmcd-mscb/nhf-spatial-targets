# Configurable Datastore and Workspace Refactor

**Issue:** #13
**Date:** 2026-03-18
**Status:** Approved

## Problem

Raw downloaded data (gridded source datasets covering CONUS) is fabric-independent, but the current workspace model couples raw data storage to a specific run directory. This means:

- Raw data cannot be shared across workspaces that use different fabrics
- Downloading CONUS-scale data to the default local drive is impractical for large datasets
- The `data/raw/` skeleton pre-creates directories for all 14 catalog sources, including 4 superseded and 1 unimplemented source
- HPC users need shared group-writable directories, which the current code does not support

## Design

### Overview

Separate the shared raw data store (fabric-independent) from the workspace (fabric-dependent). Introduce a `validate` command as a preflight gate between workspace creation and pipeline execution.

```
nhf-targets init --workdir /data/runs/my-run
  -> creates skeleton + config.yml template

# user edits config.yml (fabric path, feature ID, datastore path)

nhf-targets validate --workdir /data/runs/my-run
  -> full preflight check, writes fabric.json on success

nhf-targets fetch merra2 --workdir /data/runs/my-run
  -> checks fabric.json exists, downloads to datastore
```

### Section 1: CLI Changes

- `init` takes only `--workdir <path>`. Creates the directory with: `config.yml` (template), `.credentials.yml` (template), `data/aggregated/`, `targets/`, `logs/`.
- `validate` takes `--workdir <path>`. Reads `config.yml`, runs all preflight checks, writes `fabric.json` and `manifest.json` on success.
- `fetch`, `run`, and any aggregate/target commands check for `fabric.json` at startup. If missing, error with "run `nhf-targets validate` first".
- All pipeline commands (`fetch`, `run`, aggregate, target) change `--run-dir` to `--workdir` for consistency.
- Remove `--fabric`, `--id`, `--version`, `--buffer` flags from `init`.
- Remove the symlink-to-prior-run logic entirely.
- The `--workdir` path *is* the workspace directly (no generated run ID subdirectory).

### Section 2: config.yml Template

Generated by `init`, the user fills in before running `validate`:

```yaml
# nhf-spatial-targets workspace configuration

# ---------------------------------------------------------------------------
# Fabric
# ---------------------------------------------------------------------------
fabric:
  path: ""          # path to fabric GeoPackage (e.g., /data/gfv1.1.gpkg)
  id_col: nhm_id    # feature ID column in the fabric
  crs: EPSG:4326
  buffer_deg: 0.1   # degrees to buffer fabric bbox for source downloads
  # Optional: subset to specific HRU IDs for testing
  # subset_ids: [1, 2, 3]

# ---------------------------------------------------------------------------
# Shared raw data store (fabric-independent)
# ---------------------------------------------------------------------------
datastore: ""       # path to shared raw data directory (e.g., /mnt/bigdrive/nhf-raw)

# ---------------------------------------------------------------------------
# Directory permissions (Unix only; ignored on Windows)
# ---------------------------------------------------------------------------
# Octal string parsed as octal (e.g., "2775" = setgid + group rwx + other rx).
dir_mode: "2775"

# ---------------------------------------------------------------------------
# Spatial aggregation (gdptools)
# ---------------------------------------------------------------------------
aggregation:
  engine: gdptools
  method: area_weighted      # area-weighted mean for intensive variables

# ---------------------------------------------------------------------------
# Output
# ---------------------------------------------------------------------------
output:
  format: netcdf
  compress: true

# ---------------------------------------------------------------------------
# Targets
# ---------------------------------------------------------------------------
targets:

  runoff:
    enabled: true
    sources:
      - nhm_mwbm
    time_step: monthly
    period: "1982-01-01/2010-12-31"
    prms_variable: basin_cfs
    range_method: mwbm_uncertainty
    output_file: runoff_targets.nc

  aet:
    enabled: true
    sources:
      - nhm_mwbm
      - mod16a2_v061
      - ssebop
    time_step: monthly
    period: "2000-01-01/2010-12-31"
    prms_variable: hru_actet
    range_method: multi_source_minmax
    output_file: aet_targets.nc

  recharge:
    enabled: true
    sources:
      - reitz2017
      - watergap22d
    time_step: annual
    period: "2000-01-01/2009-12-31"
    prms_variable: recharge
    range_method: normalized_minmax
    normalize: true
    normalize_period: "2000-01-01/2009-12-31"
    output_file: recharge_targets.nc

  soil_moisture:
    enabled: true
    sources:
      - merra2
      - ncep_ncar
      - nldas_mosaic
      - nldas_noah
    time_step:
      - monthly
      - annual
    period: "1982-01-01/2010-12-31"
    prms_variable: soil_rechr
    range_method: normalized_minmax
    normalize: true
    normalize_by: calendar_month
    output_file: soil_moisture_targets.nc

  snow_covered_area:
    enabled: true
    sources:
      - mod10c1_v061
    time_step: daily
    period: "2000-01-01/2010-12-31"
    prms_variable: snowcov_area
    range_method: modis_ci
    ci_threshold: 0.70
    output_file: sca_targets.nc
```

- `fabric.path`, `fabric.id_col`, and `datastore` are required. `validate` errors if blank.
- `fabric.buffer_deg` defaults to `0.1` degrees. Used by `validate` to compute `bbox_buffered` in `fabric.json`.
- `dir_mode` defaults to `2775`, optional. Parsed as octal via `int(val, 8)`. Ignored on Windows.
- The template includes the full targets block from the current `config/pipeline.yml`.

### Section 3: workspace.py -- Path Resolver and Directory Helper

A module that all fetch/aggregate/target modules use instead of constructing paths themselves:

```python
workspace.load(workdir: Path) -> Workspace
  # reads config.yml + fabric.json, errors if not validated

workspace.raw_dir(source_key: str) -> Path
  # returns <datastore>/<source_key>/

workspace.aggregated_dir() -> Path
  # returns <workdir>/data/aggregated/

workspace.targets_dir() -> Path
  # returns <workdir>/targets/

workspace.make_dir(path: Path) -> Path
  # mkdir + apply dir_mode on Unix, plain mkdir on Windows
```

- `load()` checks for `fabric.json`. If missing, raises an error telling the user to run `validate`.
- All path construction is centralized here. Fetch modules never build `data/raw/...` paths themselves.
- `make_dir()` is the single place where Unix permissions are applied. On Windows, it creates directories with default permissions (NTFS ACLs are managed at the OS level).

### Section 4: validate Command

Runs these checks in order, failing fast on the first error:

1. **Config completeness** -- `fabric.path`, `fabric.id_col`, `datastore` are non-empty.
2. **Fabric exists** -- file at `fabric.path` exists and is readable.
3. **Feature ID column** -- `fabric.id_col` exists in the fabric GeoPackage's attribute table.
4. **Fabric metadata** -- compute SHA-256, bounding box (using `fabric.buffer_deg`), HRU count.
5. **Datastore** -- create directory if missing; create source subdirectories (only `status: current` sources with implemented fetch modules) via `workspace.make_dir()`.
6. **Credentials** -- `.credentials.yml` exists in workdir; check Earthdata credential fields are present and non-empty (no network authentication call; local check only).
7. **Catalog consistency** -- sources referenced in `variables.yml` exist in `sources.yml`, no broken references.

On success:
- Writes `fabric.json` to the workdir (the "validated" marker).
- Writes `manifest.json` with initial workspace metadata (workspace path, fabric metadata, nhf_spatial_targets version, created_utc, empty `sources` and `steps` dicts).
- Prints a summary of the workspace.

**fabric.json schema:**

```json
{
  "path": "/data/gfv1.1.gpkg",
  "sha256": "abc123...",
  "crs": "EPSG:4326",
  "id_col": "nhm_id",
  "hru_count": 109951,
  "bbox": {"minx": -125.0, "miny": 24.0, "maxx": -66.0, "maxy": 50.0},
  "bbox_buffered": {"minx": -125.1, "miny": 23.9, "maxx": -65.9, "maxy": 50.1},
  "buffer_deg": 0.1
}
```

**Datastore source directories created (current + implemented):**

- `ncep_ncar`

### Section 5: Fetch Module Changes

Each implemented fetch module updated to:

1. Accept a `Workspace` object (from `workspace.load()`) instead of a raw `run_dir: Path`.
2. Use `workspace.raw_dir(source_key)` for download paths.
3. Read bbox from `workspace.fabric` (loaded from `fabric.json`) instead of reading `fabric.json` directly.
4. Use `workspace.make_dir()` for any subdirectory creation.

Download logic, consolidation, and manifest tracking stay the same.

**Affected modules:** `merra2.py`, `nldas.py`, `ncep_ncar.py`, `modis.py`, `pangaea.py`, `reitz2017.py`

**Not affected (stubs):** `ssebop.py`, `sciencebase.py` -- these raise `NotImplementedError` and will adopt the `Workspace` interface when implemented.

### Section 6: Removals

**Removed:**
- `init_run.py` symlink-to-prior-run logic (`_reusable_raw_data()` and related code)
- `--fabric`, `--id`, `--version`, `--buffer` CLI flags from `init`
- Run ID generation logic (`YYYY-MM-DD_label_vX.Y.Z` naming)
- Pre-creation of directories for superseded/unimplemented sources
- `data/` placeholder directory in repo root

**Retained:**
- `manifest.json` provenance tracking (per-workspace)
- Atomic writes pattern in fetch modules
- `config/pipeline.yml` as a reference/template (its content informs the generated `config.yml`)

## Post-Implementation Note (2026-03-19)

**Naming clarification:** The "workspace" concept implemented here is more accurately a **project** directory. The key distinction:

- **Project** (fabric-specific): config, fabric metadata, aggregated outputs, targets, weights, logs. Created by `init`, tied to one fabric.
- **Datastore** (shared, fabric-independent): raw/consolidated source data under `<source_key>/` directories. Reusable across projects pointing to different fabrics.

Multiple projects can share one datastore. If data has already been fetched for one project, another project pointing to the same datastore will find and reuse it. The datastore path must be explicitly set in `config.yml` and should be a separate directory from the project.

Issue #38 tracks renaming `--workdir` → `--project-dir` throughout the CLI and codebase.
